正则表达式总结整理

语法

非打印字符

字符 描述 举例
\cx x 的值必须为 A-Z 或 a-z 之一。否则，将 c 视为一个原义的 ‘c’ 字符。 \cM 匹配一个 Control-M 或回车符
\f 匹配一个换页符 等价于 \x0c 和 \cL
\n 匹配一个换行符 等价于 \x0a 和 \cJ
\r 匹配一个回车符 等价于 \x0d 和 \cM
\s 匹配任何空白字符，包括空格、制表符、换页符等等 等价于 [ \f\n\r\t\v]
\S 匹配任何非空白字符 等价于 [^ \f\n\r\t\v]
\t 匹配一个制表符 等价于 \x09 和 \cI
\v 匹配一个垂直制表符 价于 \x0b 和 \cK

特殊字符

特殊字符 描述 举例
\$ 匹配输入字符串的结尾位置；如果设置了 RegExp 对象的 Multiline 属性，则 \$ 也匹配 ‘\n’ 或 ‘\r’。 要匹配 \$ 字符本身，请使用 \$
() 匹配一个子表达式的开始位置和结束位置 要匹配这些字符，请使用 ( 和 )
* 匹配前面的子表达式 0 次或多次 要匹配*字符，请使用 *
+ 匹配前面的子表达式 1 次或多次 要匹配+字符，请使用 +
. 匹配除换行符\n 之外的任何单字符 要匹配.字符，请使用 .
[ 标记一个中括号表达式的开始 要匹配[字符，请使用 [
? 匹配前面的子表达式 0 次或 1 次 要匹配?字符，请使用 \?
\ 将下一个字符标记为或特殊字符、或原义字符、或向后引用、或八进制转义符 ‘n’ 匹配字符 ‘n’。’\n’ 匹配换行符。序列 ‘\‘ 匹配 “\”，而 ‘(‘ 则匹配 “(“
^ 匹配输入字符串的开始位置；但在方括号表达式中使用时，表示不接受该字符集合 要匹配^字符，请使用 \^
{ 标记限定符表达式的开始 要匹配{字符，请使用 \^{
指明两项之间的一个选择 要匹配 字符，请使用 \^

限定符

字符 描述 举例
* 匹配前面的子表达式 0 次或多次 zo* 匹配 z zo zoo 等
+ 匹配前面的子表达式 1 次或多次 zo+ 匹配 zo zoo zooo 等
? 匹配前面的子表达式 0 次或 1 次 zo? 匹配 z zo
{n} 匹配 n 次(n 为非负整数) o{2} 匹配 food
{n,} 至少匹配 n 次(n 为非负整数) o{2,} 匹配 food foood 等
{n,m} 至少匹配 n 次且最多匹配 m 次(m >= n 为非负整数) “o{1,3}” 将匹配 “fooooood” 中的前三个 o

若要匹配一行文本开始处的文本，请在正则表达式的开始使用 ^ 字符。不要将 ^ 的这种用法与中括号表达式内的用法混淆。 若要匹配一行文本的结束处的文本，请在正则表达式的结束处使用 \$ 字符。

*、+限定符都是贪婪的，因为它们会尽可能多的匹配文字，只有在它们的后面加上一个?就可以实现非贪婪或最小匹配。

例子：

  • /<.*>/ 匹配从开始小于符号 (<) 到关闭 H1 标记的大于符号 (>) 之间的所有内容
  • /<.*?>/ 只匹配

  • /<\w+?>/只匹配开始的 H1 标签

定位符

字符 描述 举例
^ 匹配输入字符串开始的位置 如果设置了 RegExp 对象的 Multiline 属性，^ 还会与 \n 或 \r 之后的位置匹配
\$ 匹配输入字符串结尾的位置 如果设置了 RegExp 对象的 Multiline 属性，\$ 还会与 \n 或 \r 之前的位置匹配
\b 匹配每一个单词边界，即字与空格间的位置
\B 非单词边界匹配

例子：
Chapter /\bCha/ 匹配 Cha
Chapter /ter\b/ 匹配 ter

元字符

字符 描述 举例
? 当该字符紧跟在任何一个其他限制符 (*, +, ?, {n}, {n,}, {n,m}) 后面时，匹配模式是非贪婪的 对于字符串 “oooo”，’o+?’ 将匹配单个 “o”，而 ‘o+’ 将匹配所有 ‘o’
(?:pattern) 匹配 pattern 但不获取匹配结果 industr(?:y ies)就是一个比 ‘industry industries’ 更简略的表达式
(?=pattern) 在任何匹配 pattern 的字符串开始处匹配查找字符串 “Windows(?=95 98 NT 2000)”能匹配”Windows2000”中的”Windows”，但不能匹配”Windows3.1”中的”Windows”
\num 匹配 num，其中 num 是一个正整数 ‘(.)\1’ 匹配两个连续的相同字符
\s 匹配任何空白字符，包括空格、制表符、换页符等等 [ \f\n\r\t\v]

常用方法

match

str.match(patt1) 返回一个数组，实例中的数组包含 5 个元素，索引 0 对应的是整个字符串，索引 1 对应第一个匹配符（括号内），以此类推。

  • 第一个括号子表达式捕获 Web 地址的协议部分。该子表达式匹配在冒号和两个正斜杠前面的任何单词。

  • 第二个括号子表达式捕获地址的域地址部分。子表达式匹配 : 和 / 之后的一个或多个字符。

  • 第三个括号子表达式捕获端口号（如果指定了的话）。该子表达式匹配冒号后面的零个或多个数字。只能重复一次该子表达式。

  • 最后，第四个括号子表达式捕获 Web 地址指定的路径和 / 或页信息。该子表达式能匹配不包括 # 或空格字符的任何字符序列。

注意：
区分^、[^xyz]、[^a-z]的区别！！！[]中^表示负值字符集合。

使用JSON web token对用户进行跨域认证

当使用一个 API 时，其中一个挑战就是认证（authentication）。

Cookie

在传统的 web 应用中，服务端成功的返回一个响应(response)依赖于两个事件。

  • 通过存储机制保存会话信息(session)。每个会话都有它独特的信息（id），它被用来让未来的请求检索信息。
  • 其次，包含在响应头里面的信息使客户端保存了一个 Cookie。服务器自动的在每个自请求里加上了会话 id，使得服务器可以通过检索 seesion 中的信息来辨别用户。

JSON web token

根据用户请求，服务端根据用户 id 生成具有用户唯一标识的加密 token，并存储在用户信息表中。
在用户登录后返回给用户，用户将其存储在 localStorage 中。
在之后 API 请求中，在请求 header x-acess-token 中添加 token 信息。
服务端获取 token 信息后，首先解密处 token 信息，获得过期时间，是否为管理员用户等，进而进行相应操作。

web性能优化

计划与指标

  1. 比你最快的竞争对手快 20%
    根据一项心理学研究，如果希望用户感觉到你们的网站用起来比竞争对手快，那么需要比他们快至少 20%。
    研究你的主要对手，收集他们的网站在移动和桌面设备上的性能指标，确定超越他们的最低要求。

  2. 在特定限速环境下测试收集数据
    移动设备上，运行速度应该会比在桌面设备上慢 4-5 倍。移动设备上的解析时间通常要比桌面设备长 36%。
    我们可以通过网络限速以及 CPU 限速在电脑上模拟移动端体验。

    为了能够复现真实用户的浏览器配置，在有条件的情况下可以模拟以下两类环境： 1.没有安装任何插件、应用、防火墙等的电脑环境； 2.用户可能会经常安装的各类软件和使用的浏览器、浏览器插件等的电脑环境；

    在模拟限速时，为了更准确的复现网络限速环境，可以使用 devtools 以外的工具：

    • Mac OS 可以用 Network Link Conditioner(https://nshipster.com/network-link-conditioner/)；
    • Windows 可以用 Windows Traffic Shaper；
    • Linux 可以用 netem

  3. 首次加载指标
    移动端应用：
    首次绘制时间 1 秒以内，TTI（交互时间） < 5s（3G），关键文件大小 < 170KB（gzip 压缩后）

资源优化

  1. 请求资源优化
    除了常用的 gzip 压缩优化，还可以尝试 Brotli 或 Zopfli。
    Brotli 只支持 https 站点，并且当资源被压缩到极致时会变得缓慢，因此要考虑收益。
    也可以考虑使用 Zopfli。任何普通的 Gzip 压缩资源都可以通过 Zopfli 改进的 Deflate 编码达到比 Zlib 的最大压缩率小 3%到 8%的文件大小。问题是压缩文件大约需要耗费 80 倍的时间。这就是为什么在资源上使用 Zopfli 的原因。因为这些资源不会发生太大的变化，只需压缩一次但可以下载多次。

  2. 使用响应图像和 WebP
    尽量使用带有 srcset、sizes 属性的响应式图片和元素响应式图片。
    还可以通过在原生上使用 WebP 图片以及回退到 JPEG 的机制或者使用协议内容的方式中使用 WebP 格式。

  3. 图像优化
    当首次加载页面时，特定图片必须加载得更快，可以对其进行一些优化。

  4. 视频优化
    重复大量动画用动态 WebP(GIF 作为回退)来代替 video 标签。

构建优化

  1. 确定优先级
    列出全部的静态资源清单（JavaScript、图片、字体、第三方脚本以及页面上的大模块：例如轮播图、复杂的信息图表和多媒体内容），并将它们分组。

  2. 解析 JavaScript 是耗时的，所以应设置体积小的 JavaScript 资源
    可以用一些工具来分析 js 打包及依赖联系：
    webpack-bundle-analyzer;
    Source Map Explorer;
    Bundle Buddy;
    Webpack size-plugin;
    考虑使用 webpack-libs-optimization 删除未使用的方法和 polyfill。
    考虑使用一些轻量级库来替代重型库，例如用 date-fns 代替 moment.js，从 moment.js 到 date-fns 的转换可能使 3G 和低端手机上的首次绘制时间减少大约 300ms。

  3. 摇树和代码分割
    摇树的核心思想是在执行路径中未使用的任何代码都将从源中删除，减少 js 体积。
    代码拆分是另一个 webpack 功能，它将代码库拆分为按需加载的“块”。

  4. 删除未使用的、多余的代码
    通过使用 devtools 的 coverage 工具（more tools 里）查看代码覆盖使用率；
    通过 jsinspect 工具查看项目中重复的 js 代码，从而进行合适的优化。

  5. 客户端渲染还是服务端渲染
    服务器端渲染的 HTML 可能更快。

    6.设置 HTTP 缓存标头
    仔细检查是否正确设置 expires、max-age、cache-control 和其他 HTTP 缓存头。
    使用专为指纹静态资源设计的 cache-control: immutable，以避免重新验证。

交付优化

  1. 是否所有额 JS 库都采用了异步加载？
    请求页面时，浏览器获取 HTML 并构造 DOM，然后获取 CSS 并构造 CSSOM，然后通过匹配 DOM 和 CSSOM 生成渲染树。一旦出现需要解析的 JS，浏览器将停止渲染，直到 JS 被解析完成，从而造成渲染延迟。

  2. 使用 IntersectionObserver 加载大型组件

  3. 渐进式加载图片

  4. 首屏渲染关键 CSS

  5. 使用 service workers 进行缓存

  6. 优化 css 渲染性能
    使用 will-change: transform 来代替开启动画 transform: translate3D()；
    transform 代替 top、left 等属性变化的动画；
    为 opacity 的元素添加 pointer-events: none 的属性。

  7. 优化白屏渲染体验
    文字排版占位符，loading 动画添加。

  8. 优化 JS 交互代码
    通过 devtools 的 performance 检测那些耗性能的代码段，从而尝试对其优化。
    在浏览器 resize 时考虑是否需要加入节流函数等优化方式。

网络优化

  1. 迁移到 HTTPS，然后启用 HTTP/2
    随着谷歌推进更安全的 web 并最终所有的 HTTP 页面都被 chrome 视为不安全，向 HTTP/2 环境转变已经不可避免。

  2. 是否优化了各个 API 的性能？
    测试每个接口的请求速度，分析是否有可减少的请求次数。
    如果许多资源需要来自 API 的数据，API 可能会成为性能瓶颈。GraphQL 为这些问题提供了高性能的解决方案。

  3. 接口请求过程是否过长
    通过 devtools 的 network 工具，可以通过 timeline 来测试接口的请求时间，从而从中找出过程中哪一步骤可能寻在过多的耗时过程。

利用Access-Control-Allow-Origin响应头解决跨域请求原理

W3C 提供了跨域请求的标准方案（Cross-Origin Resource Sharing）。
IE8、Firefox 3.5 及其以后的版本、Chrome 浏览器、Safari 4 等已经实现了 Cross-Origin Resource Sharing 规范，实现了跨域请求。
在服务器响应客户端的时候，带上 Access-Control-Allow-Origin 头信息。

如果设置 Access-Control-Allow-Origin:*，则允许所有域名的脚本访问该资源。
Access-Control-Allow-Origin:http://www.phpddt.com.com,允许特定的域名访问。

Cross-Origin Resource Sharing 协议介绍

传统的 Ajax 请求只能获取在同一个域名下面的资源，但是 HTML5 打破了这个限制，允许 Ajax 发起跨域的请求。
浏览器是可以发起跨域请求的，比如你可以外链一个外域的图片或者脚本。但是 Javascript 脚本是不能获取这些资源的内容的，它只能被浏览器执行或渲染。

Javascript 先发出跨域请求，
然后检查回复的‘Access-Control-Allow-Origin’头。
如果这个头允许该外域访问，则 Javascript 可以读取这个回复，否则就被禁止访问。
如果请求不是一个简单的 COR，则向外域服务器发送预检验请求，如果回复的头部允许访问，则发送跨域请求，否则禁止。

下面的 Javascript 会发出预检验请求和真实请求：

由于“Content-type: text/html”不是一个简单的头，它会先向”http://bar.org/b"发出一个OPTIONS的HTTP请求。 回复可能包含这样的头：

Access-Control-Allow-Origin: http://foo.org

Access-Control-Max-Age: 3628800

Access-Control-Allow-Methods: GET，PUT, DELETE

Access-Control-Allow-Headers: content-type

“Access-Control-Allow-Origin”表明它允许”http://foo.org"发起跨域请求

“Access-Control-Max-Age”表明在 3628800 秒内，不需要再发送预检验请求，可以缓存该结果

“Access-Control-Allow-Methods”表明它允许 GET、PUT、DELETE 的外域请求

“Access-Control-Allow-Headers”表明它允许跨域请求包含 content-type 头

如果预检验请求获得通过，接下来 Javascript 就会发起真实的 COR 请求，过程跟简单的 COR 请求类似。
